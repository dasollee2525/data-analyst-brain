## 이 상황은 언제 오는가

- PM/기획자가 "우리 팀 성과를 어떻게 측정하면 좋을까요?" 라고 물어올 때
- OKR 시즌에 "어떤 KR을 써야 할까" 논의
- 신규 기능/캠페인 출시 전 "이걸 어떻게 평가하지"
- 기존 지표가 의미없다는 게 느껴질 때

## 제일 먼저 할 질문

1. **이 팀/기능이 만들어내려는 가치가 무엇인가?**
   - 지표가 먼저가 아니라 가치 정의가 먼저
   - "우리가 잘 하고 있다면 어떤 모습인가?"

2. **최종 Output은 무엇이고, 그걸 만드는 Input은 무엇인가?**
   - [[Input-Output 지표 구조]] 프레임 적용

3. **이 지표를 보고 실제로 행동이 달라지는가?**
   - [[허수 지표 판별법]] 체크리스트 적용

4. **누가 이 지표를 어떤 주기로 볼 것인가?**
   - 일간 대시보드 지표 vs 월간 리뷰 지표는 다르게 설계

## 쓸 수 있는 방법론

- [[Input-Output 지표 구조]]: 행동 가능한 지표 계층 설계
- [[허수 지표 판별법]]: 제안된 지표 검증

## 설계 순서

```
1. 가치 정의 ("무엇을 만드는 팀인가")
2. Output 지표 선정 (결과, 1~2개)
3. Input 지표 선정 (행동, 3개 이하)
4. 허수 지표 체크
5. 측정 가능성 확인 (데이터 있는가)
6. 기준값 설정 (현재 수준 파악)
```

## 피해야 할 함정

- 요청자가 원하는 답을 지표로 만들어주는 것 ("잘 하고 있어 보이게")
- 측정 가능한 것만 지표로 삼는 것 (중요한 것이 측정 불가일 수 있음)
- 너무 많은 지표 → 집중력 분산
- 지표만 설계하고 기준값 없이 끝내는 것
